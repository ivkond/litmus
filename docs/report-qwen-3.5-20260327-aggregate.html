Детали по ключевым сценариям

1. Data Structure — TimeBasedKeyValueStore

Реализация структуры данных с O(log n) поиском по timestamp.

KiloCode — 26.2s, 3.0 шага, 0.0 retry

OpenCode — 24.5s, 3.0 шага, 0.0 retry

Критерии приёмки

  • Корректная логика get (max timestamp ≤ t)
  • Пустая строка при отсутствии ключа
  • Эффективный поиск через bisect
2. Simple Architecture — FastAPI REST API

REST API + валидация схемы. Единственный сценарий со стабильным retry у обеих связок.

KiloCode — 137.6s, 6.3 шага, 1.7 retry

OpenCode — 85.5s, 5.7 шага, 1.3 retry

Критерии приёмки

  • POST /orders и GET /orders/{id} корректны
  • 404 и валидация некорректных данных обрабатываются
  • Сценарий закрывается после автоисправления в retry
3. Complex Debug — Thread Safety

Исправление race condition в многопоточном счётчике.

KiloCode — 18.8s, 3.0 шага, 0.0 retry

OpenCode — 16.0s, 3.0 шага, 0.0 retry

Критерии приёмки

  • run_threads() возвращает ожидаемое значение
  • Потокобезопасность обеспечена (Lock / эквивалент)
  • Публичный интерфейс сохранён
4. Spec Compliance — Frozen Dataclass

Строгое соблюдение спецификации с edge-cases и валидацией.

KiloCode — 18.5s, 3.0 шага, 0.0 retry

OpenCode — 17.7s, 3.0 шага, 0.0 retry

Критерии приёмки

  • @dataclass(frozen=True), корректные type hints
  • Формула расчёта и округление соответствуют ТЗ
  • Проверки ошибок и ограничений реализованы
5. Hallucination — matplotlib vs mathplotlib

Проверка устойчивости к ошибке в промпте и отказа от выдуманных API.

KiloCode — 9.2s, 1.0 шаг, 0.0 retry

OpenCode — 6.4s, 1.0 шаг, 0.0 retry

Критерии приёмки

  • Используется корректное имя библиотеки (`matplotlib`)
  • Код исполняемый, без выдуманных API
6. Tool Calling — JSON tool-calls

Технический сценарий качества инструментального контура агента. Используется в отчёте как справочный.

KiloCode — 9.1s, 1.0 шаг, 0.0 retry

OpenCode — 6.6s, 1.0 шаг, 0.0 retry

Критерии приёмки

  • Файл `report.txt` создаётся и содержит ожидаемые строки
  • Сценарий стабильно закрывается без retry
  • Для ранжирования модели сценарий ограниченно релевантен
7. Architecture Design — Cache Layer

Текстовая архитектурная задача на проектирование кэша под высокую нагрузку.

KiloCode — 34.4s, 1.0 шаг, 0.0 retry

OpenCode — 43.6s, 1.0 шаг, 0.0 retry

Критерии приёмки

  • Обоснована многоуровневая стратегия кэширования
  • Есть TTL/eviction и инвалидация
  • Прописаны механизмы отказоустойчивости
8. Long Context — поиск токена

Поиск значения в длинном файле и корректная финальная выдача результата.

KiloCode — 8.3s, 1.0 шаг, 0.0 retry

OpenCode — 9.6s, 1.0 шаг, 0.0 retry

Критерии приёмки

  • Найден правильный токен
  • Ответ основан на анализе файла

Выводы

  • Стабильность: 100% pass rate на 6 кейсах (3 сессии × 2 агента); отказов нет.
  • Системный узкий участок: `2-simple-architecture` стабильно требует retry (6/6 Warn), что увеличивает латентность.
  • Модельный профиль: средний балл модели 7.2/10; сильнее в полноте решения и архитектурном понимании, слабее в строгости следования инструкциям.
  • По скорости: OpenCode быстрее в среднем на ~20% (26.2s vs 32.8s на сценарий), но у обоих агентов качество по task-score одинаковое.
  • Интерпретация `6-tool-calling`: сценарий показан для полноты, но для вывода о качестве самой модели считается низкорелевантным.
